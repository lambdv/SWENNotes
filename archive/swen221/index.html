interface Animal{
	 String name();
	 Point2D location();
	 void location(Point2D location); //new method: a setter!
	 double speed();
	 void draw(Graphics2D g);
	
	 default void run(Point2D dest){
		 double s = this.speed();
		 double diffX = dest.x() - location().x();
		 double diffY = dest.y() - location().y();
		 double distance= Math.sqrt(diffX * diffX + diffY * diffY);
		 if (distance < s){ this.location(dest); return; } //acceptable in 1 line
		 double dir = Math.atan2(diffY,diffX);
		 this.location(new Point2D(
		 location().x() + s * Math.cos(dir),
		 location().y() + s * Math.sin(dir)
	 ));
	 }
}

    • .equals: formal properties
      • It is reflexive: for any non-null reference value x, x.equals(x) should return true.
      • It is symmetric: for any non-null reference values x and y, x.equals(y) should return true if and only if y.equals(x) returns true
      • It is transitive: for any non-null reference values x, y, and z, if x.equals(y) returns true and y.equals(z) returns true, then x.equals(z) must return true
      • It is consistent: for any non-null reference values x and y, multiple invocations of x.equals(y) consistently return true or consistently return false, provided no information used in the equals comparisons on the objects is modified
      • For any non-null reference value x, x.equals(null) should return false
    • .hashCode: formal properties
      • When invoked on the same object more than once during an execution of a Java application, the hashCode method must consistently return the same integer, provided no information used in equals comparisons on the object is altered. The result need not remain consistent from one execution of an application to another execution of the same application
      • If two objects are equal according to the equals(Object) method, then calling the hashCode method on any of the two objects must produce the same integer result.
      • Note: if two objects are not equal according to method equals(Object), it is not required that calling the hashCode method on each of the two objects must produce distinct integer results.   • Garbage Collection
    • Java memory model

      C++ memory model

      Originally, for example in C++,
      objects where implemented as ‘structs’ on the heap: a contiguous unit
      of memory that is allocated in a specific address in memory, and that
      address is used to identify the object. That memory would be ‘freed’
      only when the object can be discarded. The object identity was then just the memory address of the object

      this is NOT java’s memory model

      Java memory model

      • In Java, the identity of the object is a more abstract concept and does not correspond to the location in memory of the object
      • object address in memory cannot be accessed by the programmer

        its private secret information

      • object address in memory can be changed during the life time of the object

        In practice, this happens a lot and it is one of the main ways Java memory (de)allocation is MUCH MUCH faster then the C++ one
        can even be faster than an if statement

    • The future of Java

      Even Better memory management and optimizations

      • Valhalla: Unification of primitive types and objects
        • In Valhalla, they want to push the line even further
          • Some new kinds of objects will simply NOT have an identity!
          • The fields of those objects will be all final.
        • So, on those objects we will not be able to use ‘==’ or ‘this.f=newValue’. //or == would not be identity based
        • With those extra restrictions, it is indistinguishable if those objects are passed around/stored by reference or by value. That is, the JIT (Just in time compiler) can optimize the code by either passing/storing the memory reference of those objects OR just copying the bits of their representation.
        • This is a BIG DEAL!
        • Have you heard of Rust, and how it is ‘sort of’ as safe as java and as fast as C?
        • Rust (as C/C++) allows programmers to decide when an object is passed by pointer/reference or by copying/storing the value in place.
        • Most of the Rust performance over Java comes from Rust programmers doing the exceptionally hard and complex manual work of deciding where to do one or the other
        • Especially with abstract/generic code this is so hard because often such decision should depend on how the code is used and not on how it is structured internally.
        • Java Valhalla may be able to automate all of that, and to use the JIT (that knows about code usages) to decide how to execute the code logic.

      • Mark ‘n Sweep Algorithm

        Marking Phase

        mark all reachable objects in the program by traversing through all the objects (eg depth first search) starting from object roots

        • roots are local variables and static fields

        Mark ‘n Sweep: Sweeping Phase

        move/shuffle objects that are reachable to the left while the unreachable objects gather to the right, values after a certain point are ready to be deleted

        • this is something c/c++ can not do because in those languages, objects stay where they are born while in java objects location in memory can move during execution
        • this is only done when you really need to because its laborious

          you wait enough until you have enough unreachable objects where this process is worth effort

        • languages that only use this algorithm can cause “garbage collection pauses” which make them unsuitable for certain kind of tight real time applications
        • this is why c’s malloc function can be slow because it has to navigate complex data structures like this one to find addresses that are empty
      • Does Java use Mark ‘n Sweep?

        • there is currently competition to see which one is actually better
