          public Map<Node, List<Edge>> DjikstrasAlgorithm(Node start, Node goal){
		Map<Node, List<Edge>> backpointers = new HashMap<>(); //Map of nodes to edge
		Queue<Item> fringe = new PriorityQueue<Item>(); //Ordered by shortest "length from start to next node"
		Set<Node> visited = new HashSet<Node>();
		fringe.add(new Item(start, null, 0)); //add start to fringe
		
		while(!fringe.isEmpty()){ //until the fringe is empty
				Item item = fringe.poll(); //get item from fringe with shortest next distance
				Node node = item.node();
				Edge edge = item.edge();
				int lengthToNode = item.length();
				
				if(!visited.contains(node)){ //if the node is not visited then mark it and add to back pointers
						visited.add(node);
						backpointers.put(node, edge);
						
						if(node == goal) return ReconstructPath(start, goal, backpointers);
						
						//traverse out bound edges and add them to fringe if they are not visited
						for(Edge outEdge : node.outEdges()){ 
								Node neighbour = outEdge.to();
								if(!visited.contains(neighbour)){
										fringe.add(new Item(neighbour, outEdge, lengthToNode+outEdge.length()));
								}
						}
				}
		}
}

          Requirements

          for the shortest path outputted by the A* algorithm to be accurate, the heuristic estimate must be admissible and monotonic/consistent

          admissible

          means that we always underestimates the remaining cost

          overestimating the remaining cost puts more weight on the estimate than the actual cost, making the algorithm to take incorrect routes

          monotonic/consistent

          means that the estimate for the remaining distance for each node must have the same accuracy

          estimated path length must not get less accurate as you get closer to the goal

          can check if a heuristic is consistent, the estimate difference between node x and y should be less than or equal to the cost of x y

          Inconsistant admissible

          If the estimate is admissible, but is not consistent, then:

          • cannot commit to a node when we take it off the queue
          • may need to revisit nodes
          • no point in the visited set

          Page Rank Formula

          • Formula Explained

            First Part

            finds the probability of a user going onto your page randomly (1-d)

            divided by N because there are N nodes, this specifies the probability of the user going onto this specific node

            Second Part

            the rest of the formula is a movement multiplied/related to the value of d

            the second part is about a person moves to node A through some other edges

            b is sharing its page rank with a completely, because b is only pointing to a

            c will share its only half of its page rank with a because its pointing to both a and b, so half will be given to a and half will be given to b

            share = page rank / number of outgoing edges

            Third Part

            about nodes that have no outlinks

            this node is called a sink-node

            • its rare for an actual network to have sink-nodes

            in the case of this happening, the rank of the sink-node will be shared equally to every other node in the network

boolean isCyclicDFS(Graph graph, int v, boolean visited[], int parent) {
		visited[v] = true;
		for each neighbour n of node v {
		if (!visited[n]) {
				if (isCyclicDFS(graph, n, visited, v))
						return true;
				}

						// If the adjacent vertex is visited and is not parent of
						//current vertex, then there is a cycle in the graph
						return true;
				}
		}
		return false;
}

        • Average Bits Length
        • Generating Tree Process

          Step 1: Get frequency table of symbols

          Step 2: pop two least frequent symbols and make a parent node with the sum frequency of both and add to the queue

          Step 3: repeat step 2 until there is 1 node left in the queue, this is the root

          Step 4: assign codes to symbols by recursively adding 0 to the left branch and 1 to the right branch

          and let children inherit the starting code of parent before adding 0 or 1

          so that if parent with code c will have children with code c+0 and c+1

        • Huffman coding Pseudo Code
          //Generates the best set of codes given frequencies/probabilities 
//on all the symbols. 

Huffman(frequencies of symbols):
	1. Creates a binary tree, which is used to construct the codes.
		- Construct a leaf node (singleton tree) for each symbol.
		- Put these nodes into a priority queue, with frequency as priority.
			//(lowest frequency = highest priority)
					
		2. while there is more than one node in the queue: (i.e. > 1 tree)
			- remove the top two nodes
			- create a new tree node with these two nodes as children.
					- node frequency = sum of frequencies of the two nodes
			- add new node to the queue
				
		3. Final node is root of tree.
			- Traverse this tree to assign codes:
			- if node has code c, assign c0 to left child, c1 to right child

          Throwable

          Exception

          subtype of throwable that indicates conditions that a program might catch

          • exception and its children (apart from runtime exception) are checked

          • rare to throw and bad practice to catch exception itself

          a subtype of throwable that indicates a serious problem that programs should not catch

          • common to throw and catch error itself
          • most error are abnormal conditions

          subclass of exception that can be thrown during normal operation of the JVM

          • common to throw and catch runtime exception itself
          • its unchecked, meaning it doesn’t need to be declared in a method/constructor’s throwable clause

          Process Switching

          operating system can switch from one process to another by changing states from idle to running

          • should be avoided due to overhead
          • if os is swamped with process, system is trashed because it switches between processes without getting any work done

          Process Scheduling

          scheduler's job is to select which process to execute.

          • processes waiting to be processed are stored in the Ready Queue
          • set of all processes are stored in the Job Queue
          • set of processes waiting for an I/O device are stored in the Devices Queue
          • Processes migrate among the various queues
          • data structure is like a linked list but there is a link to the tail
          • an existing process will exist in one of these queues at a given time

      • Performance Evaluation

        Mean Squared Error

        sum of all (model output - target output) squared and divided by 2

        • commonly used as the default option
        • not good because the result is very large, making it hard to interoperate
        • sensitive to large outliers

        Root Mean Squared Error

        square root of the mean square error

        • scales the error to be the same as the target scale
        • makes error smaller
        • harder to optimize with gradient based methods because getting the differences is complex
        • not recommended

        Relative Squared Error

        total square error divided by total square error of the simple predictor

        • normalizes the error
        • used to compare model errors measured in different units
        • good model is less than 1
        • used often for linear regression

        Mean Absolute Error

        less sensitive to large outliers than mean squared error

        • Number of Weights and Biases

          number of weights in a given layer is the number of neurons in the previous layer * number of neurons in the current layer

          number of bias’s are the number of neurons in the network that are not the input layer

          HL1 weights = 3*4

          HL2 weights = 4*4

          output layer weights = 4*1

          total weights = 3*4 + 4*4 + 4*1 = 32

          total biases = 4 + 4 + 1 = 9

      • Activation Functions

        Sigmoid Function

        non-linear function that scales output to [0,1]

        • smooth s shape curve
        • output interpreted as a probability
        • suitable for binary classification

        Tanh Function

        sigmoid but scales output to [-1,1]

        • better symmetry around zero
        • used in the hidden layer for recurrent networks

        ReLU Function

        scales output by converting any value below 0 to 0 and keeping any value above 0 how it is

        • often used in the hidden layer for deep neural networks
        • widely used due to simplicity and effectiveness
        • encourages sparse representations

        Softmax Function

        exponential function that maps input to a probability distribution over a set of possible outcomes

        • often used in the output layer
        • used when output is a multi-class classification tasks
        • converts vector of values to vector of probability distribution of the original values
