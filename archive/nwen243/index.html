        • Framing Approaches

          Idea 1: flags

          we can sentinels/flags to label the start and end of a data frame with a special byte eg: , 0111 1110 for start and 0111 1110 for end

          the problem with this approach is that if the actual message itself contains the flag, then the receiver will terminate the message before its over. its also a waste of data

          we can address this by playing around with the encoding of the message by bit stuffing the end of the message with a 0, receiver removes the first 0 and the actual 0 is after

          Idea 2: header

          we could specify the length/size of the message in the header (extra information at the start) of the data frame, so the receiver can just read the specified number of bytes and then terminate

          the problem with this is that errors can happen, and if the header has an error, then it corrupted the rest of the data and its unreadable

          Idea 3: clock

          we can make each data frame a fixed length (avoid bit stuffing) and attach a clock that syncs the sender and receiver.

          the start of each data frame will also be the same so when we get another data frame we know when we are at a new frame

      • Error Detection
        • detection approaches

          Parity Check

          for every btye or number of bits, instead of storing the data for all 8 bits, we only use 7 and the last bit is reserved for a parity bit

          the parity bits should form a pattern, eg there should be even or odd parity

          if the parity check fails (expected odd number but it was even) then we know there was an error somewhere

          the problem with this is that, if there were even number of errors, then the parity check will be incorrect, because the parity check can pass but that doesn‚Äôt mean the data does not contain an error

          Checksum

          add up all the words that are transmitted and then transmit the result of that sum

          receiver preforms the same comparison and compares the provided result vs its result

          there is a chance that the sum is corrupted and the data has been corrupted to match but the chance is very low, even when accounting for the number of data we send is large

          if any transmitted data, including the checksum itself is corrupted, then the result will not match and the receiver knows that an error has occurred

          CRC

          checksum but better, uses math that makes the chance of sum and data error to match to be even lower

          not only does crc deal with errors better than checksum, it also deals with ‚Äúbursty‚Äù errors

          switching techniques

          there are 3 ways it can use the forwarding table

          1) Datagrams (connectionless switching)

          just include in every datagram enough information to enable any switch to decide how to get it to its destination

          2) virtual circuit switching (connection-oriented)

          alert connection-oriented model uses concept of virtual circuits (VC)

          it requires to set up a virtual connection from source to destination before data is sent

          • connection setup phase establishes a state at each node/switch to define a path, after path has been established, then packets can start being sent. all packets will follow the same path
            • each switch keeps track of ongoing connections in its virtual circuit table
              • each switch/node makes a VCI identifier for the path
          • data transfer phase
          • Advantages of Virtual Circuits
            • guarantees there is a route from host to dest by the time data is being sent (because connection phase goes through the network)
