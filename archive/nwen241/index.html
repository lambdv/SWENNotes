      • Function like Marco

        Problems with Function-like Macros

        macros substitute your function calls in the pre-processing phase BEFORE other operations are executed such as calculation, this can cause bugs.

        1. type casting

          when we cast the function like macro as an int before the call in code, it will not cast the function output but only the first r. this is because the function call is substitution with the body of the function, then the casting logic is applied 

          #define SQ(X) X * X

//before compile
(int)SQ(r); 
//after compile
(int)r * r; //only the first r is casted to int, not the output :(

          to fix this we need to put parenthesis around the function like macro

          #define SQ(X) X * X

//before compile
(int)(SQ(r)); 
//after compile
(int)(r * r); //r*r is preformed, then output is type casted :)
        1. calculation
          #define SQ(X) X * X

//before compile time
SQ(r1 + r2);

//after compile
r1 + r2 * r1 + r2; //r2 is multiplied by r1 THEN r1 and r2 is added

          fix

          #define SQ(X) ((X) * (X))

//before compile time
SQ(r1 + r2);

//after compile
(r1 + r2) * (r1 + r2); //r1 + r2 are getting multiplied by itself
        1. increment

          we don‚Äôt want to pass variables that are increment with ++ into macros because instead of increment once and using the incremented value in the calculation, it will increment the value EACH time it is used in the calculation

          in this case it will increment twice

          #define SQ(X) ((X) * (X))

//before compile
SQ(++r);

//after compile
((++r) * (++r)); //will increment twice

          for this reason you should not pass incremented values into macros, instead increment it before hand

              What for does

              • A process calling fork() spawns a new process (child), which is a copy of the calling process.
              • After a successful fork() call, two copies of the original code will be running.
              • After the fork(), both processes not only run the same program, but they resume execution as though both had called the system call.
              • fork() returns an integer value to both parent and child process.

              we can find whether the process is a child or parent based on the return value, if its 0 then its the child, if its the PID of the child then its the parent
